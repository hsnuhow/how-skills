## 貫穿全部來源的四個系統性陷阱

這一節是全系列最該先讀的部分。欄位名可以現抓，但下面這些只能靠踩過的人寫下來。

### 一、HTTP 200 不等於拿到資料

這是台灣政府端點最一致的失敗模式，而且形態有八種：

| 形態 | 實例 |
|---|---|
| 200 + HTML 首頁 | TPEx 未知路徑（加 `-L` 得 22,964 bytes 首頁）、CBC OpenAPI（302 → 65 KB 首頁） |
| 200 + 錯誤字串 | GCIS uuid 錯誤 → `此API不存在`；MOL resourceID 錯誤 → `success:false` |
| 200 + 空 body | GCIS 統編型 API 用 `Company_Name like` |
| 200 + 0 bytes | PCC 官方 XML 不存在的期別 |
| 200 + 98 bytes | MOPS `REPORT_ID=A`（Big5 解碼後為「檔案不存在!」） |
| 200 + **別年的正確資料** | MOF funid 對不上年份時靜默回最新年，bytes 與真檔完全相同 |
| 200 + byte-identical 的錯誤表 | TPEx `mopsfin_t187ap05_OA` 與 `_O` 的 md5 相同 |
| **203**（不是 200） | 財政部 fia `limit>1500`；`resp.ok` 與 `< 400` 都會誤放行 |

**一律驗 payload，不要驗 status code。** `scripts/twdata/_fetch.py` 已內建這層防禦
（簽章比對 + `min_bytes` 門檻），新端點請走它，不要裸 `requests.get`。

### 二、日期格式在同一機關內就不一致

沒有「某機關用民國」這種規則可套：

- **DGBAS**：失業率 `2026M05` vs 薪資 `202605`（同批 XML，兩種格式）
- **CBC**：經研處西元、業務局 `A13Rate.csv` 民國、外匯局 001–014 民國
- **TWSE**：月營收 ROC 7 碼、基本資料成立日期西元 8 碼（同一 API 平台）
- **MOPS**：`SYEAR` 吃**西元**，與 TWSE/TPEx 的 ROC 慣例相反
- **GCIS 民國 7 碼 vs fbfh 西元 8 碼**——而這兩者正是要互相 join 的

逐檔確認。

### 三、期別欄位帶隱形字元

DGBAS 薪資的 `202605Ⓟ`（初步值）、`202604Ⓡ`（修正值）是 U+24C5 / U+24C7 圈字元，
直接拿去 join **會靜默對不上**。NDC 燈號欄有尾端空白（`'黃藍 '`），
group by 原始值會把同一類別拆成兩類。

### 四、母項與子項同欄，禁止子字串比對

MOEA 的 `零售業`、`綜合商品零售業`、`百貨公司業` 全部並列在 `行業別` 一欄；
比對 `零售業` 會一次撈到 20 個業別，**數字看起來合理但錯誤**，唯一症狀是期別重複。
MOF 的 58 含 581/582 同理。一律精確相等。
