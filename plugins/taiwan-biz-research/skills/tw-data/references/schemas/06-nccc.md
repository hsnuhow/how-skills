# NCCC 財團法人聯合信用卡處理中心 開放資料

## URL 組成規則

所有檔案均為靜態 CSV，直接 GET 即可，無需 API key、無 rate limit、UTF-8-BOM 編碼、`\r\n` 換行、數值無千分位。

```
https://www.nccc.com.tw/dataDownload/{維度資料夾}/{檔名}.CSV
```

檔名依資料來源分兩系：

| 前綴 | 意義 | 檔名骨架 |
|---|---|---|
| `BANK_` | 我國 38 家發卡機構的**發卡端**資料（持卡人屬性可切） | `BANK_{地區碼}_{業別碼}_{維度碼}.CSV` |
| `NCCC_` | 聯卡中心**處理／收單端**資料（無持卡人屬性） | `NCCC_{地區碼}[_{業別碼}][_FR].CSV` |

例外骨架：
- EC（網購）：`EC/BANK_EC_{業別碼}.CSV` — **無地區維度**
- 跨境／跨縣市：`ICR|DCR/BANK_{地區碼}_ALL_{ICR|DCR}.CSV` — 業別位固定為 `ALL`
- 國外消費 Top10：`Country/BANK_{地區碼}_COUNTRY_{AMT|NO}.CSV`
- 詐欺（發卡端）：`Fraud/BANK_FRAUD_TWN_{業別碼}.CSV` — **僅全國**
- 詐欺（收單端／外國卡）：`Fraud/ACQ/FR/BANK_FRAUDACQ_{地區碼}_{業別碼}_FR.CSV`
- 業務統計年報：`dataDownload/{01..12}_{英文檔名}.csv`（**副檔名小寫**，無子資料夾）

**資料夾名含空格者需 URL-encode 成 `%20`**（例：`Age%20Group`、`NCCC%20Open%20Data/Location%20EC`）。

### 網站目錄分頁

清單頁用 IBM WCM 分頁，預設每頁 10 筆。加 `WCM_PageSize.{GUID}=600` 可一次取回全部：

```
https://www.nccc.com.tw/wps/wcm/connect/zh/home/openinformation/CreditCardData?WCM_PI=1&WCM_Page.{GUID}=1&WCM_PageSize.{GUID}=600
```

GUID 每個頁面不同，需先抓一次原始頁再以 `WCM_Page\.[0-9a-f-]{36}` 取出。母頁 `CreditCardData` 本身列的 161 筆**只是「兩性消費樣態」單一維度**（23 地區 × 7 業別），不是全站總數；其餘維度都在子頁。全站實際約 2,300 個檔案。

## 對照表三：業別碼 → 中文業別 → 實際涵蓋行業

官方定義（取自業務說明頁）：

| 業別碼 | 檔內「信用卡產業別」欄值 | 官方全名 | 實際涵蓋 |
|---|---|---|---|
| `FD` | `食` | 食品餐飲類 | 食品類、中西餐點、日韓料理、飲食小吃等 |
| `CT` | `衣` | 服飾類 | 鞋、皮件、服飾等 |
| `LG` | `住` | 住宿類 | **旅館、裝潢及電器**（注意：家電與裝潢被歸在「住」，不是百貨） |
| `TR` | `行` | 交通類 | 機車、汽車、機票等 |
| `EE` | `文教康樂` | 文教康樂 | **醫療**、文教、運動、保健、娛樂等（醫療在此，不是「其他」） |
| `DP` | `百貨` | 百貨類 | 大型綜合百貨及超級市場等 |
| `OT` | `其他` | 其他類 | 不在前述類別內之行業 |
| `ALL` | （檔內仍逐列標示 7 種業別） | 全部 | 見下方說明 |

**`ALL` 的兩種不同語意**：
1. 在 `ICR`/`DCR` 中，`ALL` 是檔名固定佔位符，代表不分業別的全體交易。
2. 在 `Gender`/`Age Group`/`Annual Income`/`Classified Employment`/`Education Level`/`Gender X *`/`EC` 中，存在**網站未列出但可下載**的 `ALL` 檔（如 `Gender/BANK_TWN_ALL_GD.CSV`）。它**不是**業別合計，而是把 7 個業別檔**縱向堆疊成一個檔**（`信用卡產業別` 欄仍逐列變動）。僅 `TWN` 有此捷徑檔，縣市層級（如 `BANK_TPE_ALL_GD.CSV`）為 404。
3. 在 `Fraud`/`Fraud/ACQ/FR` 中，`ALL` 的欄值為 `全部`，是真正的業別合計。

**業別覆蓋不齊的兩處**：`NCCC Open Data/Industry` 與 `Foreign/Industry` **只有 6 種業別，缺 `OT`**。

## 命名陷阱

**1.（重要修正）`Top10ForeignCardConsumptionCount` / `Top10ForeignCardConsumptionPrice` 是 outbound，不是 inbound。**

英文目錄名含 "ForeignCard"，極易誤讀為「外國卡在台消費」。實際上：

- 中文頁面標題為**「前十大國外消費型態」**，資料列標題為**「台北市持卡人前十大國外消費金額及筆數」**——主體是**我國持卡人**，客體是**國外消費**。
- 它掛在 `CreditCardData`（＝「我國信用卡消費資料」，我國 38 家發卡機構）之下，資料來源本身就只涵蓋台灣發行的卡。
- 檔案結構佐證：`地區` 欄是台灣縣市代碼（持卡人帳單地址），`國別` 欄是美國／日本／法國等外國——若為 inbound，地區欄該是台灣消費地、國別欄該是發卡國，但金額量級不符。
- 量級佐證：TPE 202604 Top10 合計 138.7 億，同月 `ICR`（跨境交易，定義為帳單原始幣別非新臺幣）為 123.9 億，兩者同量級；而同月全國 inbound（`Foreign/Location/NCCC_TWN_FR.CSV`）僅 80.1 億，若 Top10 是 inbound，單台北市不可能超過全國。

**真正的 inbound（外國卡在台消費）在 `NCCCData/Foreigncardconsumption`，即 `Foreign/Location`、`Foreign/Location EC`、`Foreign/Industry` 三個資料夾**，其中文標題明寫「外國持卡人以信用卡支付…」。

**2. `Crossborderconsumption`（ICR）的「跨境」是幣別定義，不是地理定義。** 官方欄位說明：「跨境交易係指帳單原始幣別不為新臺幣之交易」。因此在台灣刷卡但以外幣計價（如境外電商、DCC 動態貨幣轉換）會被計入跨境；反之在海外刷卡但以新臺幣計價則不計入。這與「國人出國消費」不完全等同。

**3. `Crossingcountyconsumption`（DCR）的「跨縣市」是郵遞區號比對。** 定義為「持卡人帳單地址郵遞區號與交易發生之特約商店郵遞區號位於不同縣市」，因此網購（商店登記地在台北）會大量污染此指標，不宜直接當「觀光／通勤外出消費」代理變數。

**4. `BANK_` vs `NCCC_` 是兩套不可混用的口徑。** `BANK_*` 為 38 家發卡機構的**發卡端**資料（可切持卡人屬性）；`NCCC_*` 為聯卡中心**處理／收單端**資料（含 UnionPay、DFS，含 Debit Card，含非會員機構交易）。兩者母體不同，不可相加，同一月份的全國總額也不會相等。

**5. 業別 `LG`（住）不只是旅宿。** 官方定義含「旅館、**裝潢及電器**」。做旅遊住宿市場分析時直接用 `LG` 會嚴重高估。同理 `EE`（文教康樂）**含醫療與保健**，不是純娛樂支出。

**6. 資料夾 `Gender` 對應目錄頁 `Sexconsumption`，而 `Billing Address` 對應 `Billing+Address`（URL 中空格為 `+`，下載路徑中為 `%20`）。** 目錄頁 slug 與 `dataDownload` 資料夾名無一致規則，必須以本文對照表一為準，不可從頁面 slug 推導檔案路徑。

**7. 詐欺型態字串在兩個資料集中寫法不一致**：發卡端 `Fraud/` 寫 `MOTO / EC`（斜線前後有空格），收單端 `Fraud/ACQ/FR/` 寫 `MOTO/EC`（無空格）。跨表 join 時需正規化。

**8. 「161 個資料集」是單一維度的檔數，不是全站總數。** 該數字來自 `CreditCardData` 母頁列出的「兩性消費樣態」23 地區 × 7 業別。全站實際檔案數約 2,300，schema 種類 15 種。
